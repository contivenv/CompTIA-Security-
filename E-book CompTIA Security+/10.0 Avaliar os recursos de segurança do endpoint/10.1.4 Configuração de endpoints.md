Se a segurança do endpoint for comprometida, é preciso considerar várias classes de vetores para a mitigação:

- **Engenharia Social**: se o malware foi executado por um usuário, utilize ações de educação e conscientização em segurança para reduzir o risco de ataques futuros. Revise as permissões para verificar se a conta poderia ser operada com um nível de privilégio inferior.
- **Vulnerabilidades**: se o malware explorou uma falha de software, instale o patch ou isole o sistema até que uma correção seja desenvolvida.
- **Ausência de Controles de Segurança**: se o ataque poderia ter sido prevenido por proteção de endpoint/antivírus, firewall de host, filtro de conteúdo, DLP ou MDM, avalie a possibilidade de implantar esses recursos no endpoint. Se isso não for viável, isole o sistema para evitar exploração pelo mesmo vetor.
- **Desvio de Configuração**: se o malware explorou uma alteração de configuração não documentada (como software shadow IT ou um serviço/porta não autorizado, por exemplo), reestabeleça a configuração baseline e revise os procedimentos de gerenciamento de configuração para evitar esse tipo de alteração ad hoc.
- **Configuração Fraca**: se a configuração foi aplicada corretamente, mas ainda assim foi explorada, revise o template para desenvolver configurações mais seguras. Certifique-se de que o template seja aplicado a hosts semelhantes.

## Controle de acesso

O controle de acesso refere-se à regulamentação e ao gerenciamento das permissões concedidas a indivíduos, softwares, sistemas e redes para acesso a recursos ou informações. Os controles de acesso garantem que apenas entidades autorizadas possam realizar determinadas ações ou acessar certos dados, enquanto entidades não autorizadas têm o acesso negado. Os conceitos de controle de acesso se aplicam a redes, acesso físico, dados, aplicativos e nuvem.

## Princípio do privilégio mínimo

A implementação do princípio do privilégio mínimo (PoLP) é fundamental para melhorar a proteção de endpoints e minimizar o risco de problemas de segurança. O princípio do menor privilégio estabelece que usuários, aplicativos e processos devem receber apenas as permissões mínimas necessárias para realizar suas funções, e nada além disso.

Existem vários métodos práticos para implementar o privilégio mínimo. Um primeiro passo essencial para implementar efetivamente o privilégio mínimo é auditar minuciosamente as funções, privilégios e responsabilidades do usuário. Esse processo permite que as organizações entendam qual acesso cada usuário precisa para desempenhar sua função de forma eficaz. Os controles e permissões de acesso podem ser ajustados para adotar um princípio de privilégio mínimo que reflita melhor os resultados da auditoria.

As ferramentas de gerenciamento de usuários e contas também são essenciais ao implementar o princípio do privilégio mínimo. A revisão e a remoção regulares de contas não utilizadas ou desnecessárias reduzem os alvos em potencial para um invasor. Da mesma forma, os privilégios temporários, que concedem direitos de acesso adicionais por um tempo limitado e apenas quando necessário, podem ajudar a manter os privilégios o mais restritivos possível.

Outra abordagem prática para restringir o acesso é usar o controle de acesso baseado em função (RBAC). O RBAC atribui direitos de acesso ao sistema com base em funções predefinidas, e cada função tem um conjunto cuidadosamente definido de permissões que correspondem aos requisitos de qualquer função específica da organização. Essa abordagem garante que os usuários tenham acesso suficiente para executar suas tarefas, mas nada mais.

O princípio do privilégio mínimo também se aplica a aplicativos de software e sistemas operacionais, não apenas aos usuários. Por exemplo, garantir que os aplicativos sejam executados com as permissões mínimas necessárias pode impedir que sejam explorados para executar ações privilegiadas.

## Listas de controle de acesso

As listas de controle de acesso (Access Control Lists, ACL) em sistemas e redes de computadores são usadas para impor políticas de controle de acesso. Uma ACL é uma lista de regras ou entradas que especificam quais usuários ou grupos têm acesso permitido ou negado a recursos específicos ou executam determinadas ações. Em redes, as ACL estão associadas a roteadores, firewalls ou dispositivos semelhantes e definem regras que determinam como o tráfego de rede é filtrado ou encaminhado com base em critérios como endereços de IP de origem, endereços de IP de destino, portas ou protocolos.

As ACL podem ajudar a controlar o acesso à rede e proteger contra atividades não autorizadas ou mal-intencionadas. As ACL controlam o acesso a arquivos, diretórios ou recursos do sistema em sistemas operacionais e sistemas de arquivos. Cada entrada de controle de acesso (Access Control Entry, ACE) normalmente contém um identificador de usuário ou grupo e permissões associados que controlam as ações permitidas ou negadas. Essas permissões geralmente incluem ler, gravar, executar e, às vezes, limites mais granulares, como modificar, excluir ou listar.

Embora as ACL ofereçam flexibilidade e controle, o gerenciamento de políticas complexas de controle de acesso com várias entradas de ACL pode se tornar desafiador. A complexidade aumenta o risco de configurações incorretas. Portanto, o planejamento adequado, as revisões periódicas e as configurações que seguem práticas recomendadas são essenciais ao implementar e manter as ACL.

## Permissões do sistema de arquivos

Com a segurança do sistema de arquivos, cada objeto nele tem uma lista de controle de acesso (Access Control List, ACL) associada. A ACL contém uma lista de contas (entidades) com permissão para acessar o recurso e as permissões que elas têm sobre ele. A ordem das ACE na ACL é importante para determinar as permissões efetivas de uma determinada conta. As ACL podem ser impostas por um sistema de arquivos compatível com permissões, como NTFS, ext3/ext4 ou ZFS.

Por exemplo, no Linux, existem três permissões básicas:

- **Read (r, leitura)**: é a capacidade de acessar e visualizar o conteúdo de um arquivo ou listar o conteúdo de um diretório.
- **Write (w, gravação)**: é a capacidade de salvar alterações em um arquivo ou criar, renomear e excluir arquivos em um diretório (também requer execução).
- **Execute (x, execução)**: é a capacidade de executar um script, programa ou outro arquivo de software, ou a capacidade de acessar um diretório, executar um arquivo a partir desse diretório ou executar uma tarefa nesse diretório, como pesquisa de arquivos.

Essas permissões podem ser aplicadas no contexto do usuário proprietário (`u`), de uma conta de grupo (`g`) e de todos os outros usuários/mundo (). Uma cadeia de permissões lista as permissões concedidas em cada um desses contextos:

`d rwx r-x r-x home`

A cadeia de caracteres acima mostra que, para o diretório (d), o proprietário tem permissões de leitura, gravação e execução, enquanto o contexto do grupo e outros usuários têm permissões de leitura e execução.

O [[comando chmod]] é utilizado para modificar as permissões de arquivos em sistemas Unix/Linux. Ele pode ser usado no modo simbólico ou no modo absoluto. No modo simbólico, o comando funciona da seguinte maneira:

`chmod g+w, o-x home`

O efeito desse comando é acrescentar a permissão de gravação ao contexto do grupo e remover a permissão de execução do outro contexto. Por outro lado, o comando também pode ser usado para substituir as permissões existentes. Por exemplo, o comando a seguir aplica a configuração mostrada na primeira cadeia de permissões, que define as permissões como rwxr-xr-x:

`chmod u=rwx,g=rx,o=rx home.`

No modo absoluto, as permissões são atribuídas usando a notação octal, onde r=4, w=2 e x=1. Por exemplo, o comando a seguir tem o mesmo efeito:

`chmod 755 home`

Neste exemplo, o 755 corresponde às permissões atribuídas ao usuário, ao grupo e aos demais, onde as permissões do usuário são representadas por 7, as permissões do grupo por 5 e as permissões dos outros também por 5. Os números são gerados somando-se os valores associados a r (leitura), w (escrita) e x (execução). A única combinação de valores que resulta em 7 é 4+2+1, ou seja, r, w, x. Da mesma forma, a única combinação de valores que resulta em 5 é 4+1, ou seja, r e x. Isso significa que o proprietário possui r, w e x, enquanto o grupo e os outros têm apenas r e x.

## Listas de permissão e de bloqueio de aplicativos

Um elemento da configuração do endpoint é a política de controle de execução que define aplicativos que podem ou não ser executados.

- Uma [[allow list]] (ou lista de aprovação) nega a execução, a menos que o processo seja explicitamente autorizado.
- Uma [[block list]] (ou lista de bloqueio) geralmente permite a execução, mas proíbe explicitamente os processos listados.

O conteúdo das listas de permissão e das listas de bloqueio precisa ser atualizado em resposta a incidentes e à busca e ao monitoramento contínuos de ameaças.

A threat hunting também pode provocar uma mudança estratégica. Por exemplo, se você emprega principalmente negações explícitas mas seus sistemas estão sujeitos a inúmeras intrusões, terá de considerar a adoção de um modelo de “privilégios mínimos” e usar uma abordagem de negação a menos que esteja listado. No entanto, esse tipo de mudança pode ser altamente disruptiva, por isso deve ser precedida por uma avaliação de risco e uma análise de impacto nos negócios.

O controle de execução também pode ser difícil de configurar de forma eficaz, com muitas oportunidades para os agentes de ameaças escaparem dos controles. A análise detalhada do ataque pode sugerir a necessidade de mudanças no mecanismo existente ou de um sistema mais robusto.

## Monitoramento

O monitoramento desempenha um papel vital no fortalecimento e hardening do endpoint, ajudando a aplicar e manter as medidas de segurança implementadas durante o processo de hardening. Quando os dispositivos já estão protegidos, o monitoramento ajuda a garantir que essas condições permaneçam vigentes.

Os analistas de segurança podem detectar mudanças que enfraquecem a configuração protegida por meio do monitoramento contínuo. Por exemplo, se uma porta anteriormente desabilitada for detectada como aberta ou um serviço desabilitado for habilitado, as ferramentas de monitoramento podem alertar os analistas sobre a mudança, que pode indicar uma violação.

Além disso, o monitoramento pode fornecer dados valiosos para fins de conformidade e auditoria. Relatórios regulares sobre o status dos dispositivos de endpoint podem verificar se os parâmetros de referência para hardening foram efetivamente implantados e mantidos, apoiando a conformidade com vários regulamentos e padrões do setor.

## Imposição de configuração

A imposição de configuração descreve os métodos usados para garantir que os sistemas e dispositivos dentro da rede de uma organização sigam as configurações de segurança obrigatórias.A imposição da configuração geralmente depende de alguns recursos importantes.

- Os **parâmetros de referência padronizados para configurações** (Standardized Configuration Baselines) são definidos por organizações como NIST, CIS ou a própria organização cliente e usados como referência para determinar como os sistemas e dispositivos devem ser configurados.
- São usadas **ferramentas de gerenciamento de configuração automatizada (Automated Configuration Management Tools)** para aplicar e manter automaticamente os parâmetros de referência padronizados para as configurações em todo o ambiente.
- O **monitoramento contínuo e as verificações de conformidade** são cruciais para detectar desvios das configurações obrigatórias.
- Os processos de **gerenciamento de mudanças (Change Management)** garantem que as mudanças de configuração sejam devidamente avaliadas, testadas e aprovadas antes da implementação.

O gerenciamento de regras de firewall na rede de uma organização é um exemplo prático de uma configuração que pode se beneficiar da imposição de configuração. Uma organização pode usar uma ferramenta de gerenciamento de configuração automatizada para garantir que os firewalls do dispositivo estejam configurados corretamente, incluindo o bloqueio de todo o tráfego de entrada por padrão e a permissão apenas de conexões específicas e necessárias. O monitoramento contínuo detecta quaisquer alterações e as reverte automaticamente para impor a configuração segura aprovada.

Além disso, as verificações regulares de conformidade garantem que os firewalls sigam as configurações aprovadas, e quaisquer mudanças precisam ser revisadas e aprovadas por meio de processos adequados de gerenciamento de mudanças antes da implementação.

## Política de grupo

Política de grupo (Group Policy) é um recurso do sistema operacional Microsoft Windows que fornece gerenciamento e configuração centralizados de sistemas operacionais, aplicativos e configurações do usuário em um ambiente do Active Directory. As Políticas de grupo impõem configurações de segurança, como aquelas determinadas por um parâmetro de referência, aplicando configurações consistentes em todos os sistemas vinculados a Políticas de grupo específicas. Em termos gerais, as Políticas de grupo são vinculadas a contêineres chamados Unidades organizacionais (Organizational Units, OU) que normalmente contêm objetos do usuário e do computador. As Políticas de grupo vinculadas à UO se aplicam a todos os objetos contidos nela.

Exemplos de configurações comuns de Política de grupo incluem políticas de senha, direitos do usuário, configurações do Firewall do Windows, configurações de atualização do sistema, restrições de instalação de software e muitos outros. A aplicação central de configurações usando a Política de grupo reduz possíveis problemas relacionados a configurações incorretas ou inconsistentes.

## SELinux

O [[SELinux]] é um recurso de segurança do kernel Linux que oferece suporte a políticas de controle de acesso, incluindo controles de acesso mandatórios (MAC). O SELinux permite um controle de permissão mais granular sobre cada processo e objeto do sistema dentro de um sistema operacional, limitando estritamente os recursos que um processo pode acessar e quais operações pode executar. O SELinux opera com base no princípio de que, se um processo ou usuário não precisa de acesso a recursos para operar, ele será bloqueado para isolar melhor os aplicativos, restringir o acesso ao sistema e aos arquivos e impedir que programas mal-intencionados ou defeituosos causem danos ao sistema. Os recursos do SELinux também estão disponíveis no sistema operacional Android: [https://source.android.com/docs/security/features/selinux](https://source.android.com/docs/security/features/selinux). Devido às diferenças significativas de arquitetura entre o Linux e o Android, o recurso SELinux no Android é possibilitado usando o SEAndroid para fornecer funcionalidade semelhante, mas usando uma base de código mantida separadamente.
