

Enquanto o balanceamento de carga distribui o tráfego entre nós de processamento independentes, o cluster permite que múltiplos nós redundantes de processamento compartilhem dados entre si para aceitar conexões. Isso fornece redundância. Se um dos nós no [[cluster]] parar de funcionar, as conexões podem executar [[failover]] para um nó de trabalho. Para os clientes, o cluster parece ser um único servidor. Um balanceador de carga distribui solicitações de cliente entre nós de servidor disponíveis em uma farm ou pool e está geralmente associado ao gerenciamento de tráfego da Web, enquanto os clusters são usados para fornecer redundância e alta disponibilidade para sistemas como bancos de dados, servidores de arquivos e outros.

## Virtual

Por exemplo, uma organização pode querer provisionar dois dispositivos de balanceamento de carga para que, se um falhar, o outro ainda possa lidar com conexões de cliente. Ao contrário do balanceamento de carga com um único dispositivo, o IP público usado para acessar o serviço é compartilhado entre as duas instâncias no cluster. Esta disposição é referida como um IP virtual ou endereço compartilhado ou flutuante. As instâncias são configuradas com uma conexão privada, na qual cada uma é identificada por seu endereço IP “real”. Essa conexão executa um protocolo de redundância, como o CARP (Common Address Redundancy Protocol), permitindo que o nó ativo “possua” o IP virtual e responda às conexões. O protocolo de redundância também implementa um mecanismo de heartbeat para permitir failover para o nó passivo se o ativo sofrer falhas.

O cluster de balanceamento de carga à esquerda é composto por um servidor virtual, um balanceador de carga redundante ativo e um balanceador de carga redundante passivo. A fazenda de servidores web à direita é composta por três servidores web. Os primeiros passos são:

1. Serviço de balanceamento de carga configurado como cluster ativo ou passivo.

2. Os nós do cluster são conectados por IPs internos e por um protocolo de redundância.

3. O nó ativo responde às solicitações no IP virtual.

## Clustering ativo/passivo (A/P) e ativo/ativo (A/A)

No exemplo anterior, se um nó estiver ativo, o outro está passivo. Isso é chamado de clustering ativo/passivo. A maior vantagem das configurações ativas/passivas é que o desempenho não é afetado negativamente durante o failover. No entanto, há custos mais altos de hardware e sistema operacional devido à capacidade não utilizada.

Um cluster ativo/ativo significa que ambos os nós estão processando conexões simultaneamente. Isso permite que o administrador use a capacidade máxima do hardware disponível enquanto todos os nós estão funcionais. No caso de um failover, a carga de trabalho do nó com falha é imediata e transparentemente deslocada para o outro nó. Nesse momento, a carga de trabalho no nó ou nós restantes é maior e o desempenho é degradado.

**Configurações N+1 e N+M:**Em uma configuração padrão ativa/passiva, cada nó ativo deve ser acompanhado por um nó passivo. Para reduzir os custos associados aos nós passivos, utilizam-se as configurações N+1 e N+M:

- **Configuração N+1:**Nesse caso, um único nó passivo é compartilhado entre vários nós ativos. Por exemplo, com cinco nós ativos, em vez de possuir cinco nós passivos, apenas um nó passivo é provisionado para assumir em caso de falha de qualquer um dos nós ativos. Essa configuração reduz significativamente o número de nós passivos necessários.
- **Configuração N+M:**Nesta configuração, vários nós passivos são compartilhados entre vários nós ativos. Por exemplo, com dez nós ativos, podem estar disponíveis dois ou três nós passivos. Esses nós passivos podem assumir em caso de falhas, oferecendo um equilíbrio entre redundância e eficiência de custos.

## Clustering de aplicativos

O clustering também é muito comumente usado para fornecer serviços de aplicativos tolerantes a falhas. Se um servidor de aplicativos sofrer uma falha no meio de uma sessão, os dados do estado da sessão serão perdidos. O clustering de aplicativos permite que os servidores no cluster comuniquem informações de sessão entre si. Por exemplo, se um usuário fizer login em uma instância, a próxima sessão poderá ser iniciada em outra instância e o novo servidor poderá acessar os cookies ou outras informações usadas para estabelecer o login.
