# 

Deve ser atribuída uma política a cada host ou roteador que usa IPsec. Uma política IPsec define o mecanismo de autenticação, bem como o uso de AH/ESP e modo de transporte ou de túnel, para uma conexão entre dois pares.

As funções de criptografia e hash do IPsec dependem de um segredo compartilhado. O segredo deve ser comunicado a ambos os pares, e estes devem realizar autenticação mútua para confirmar a identidade um do outro. O protocolo [[Internet Key Exchange (IKE) (troca de chaves na Internet)]] implementa um método de autenticação, seleciona as cifras criptográficas que são mutuamente suportadas por ambos os pares e realiza a troca de chaves. O conjunto de propriedades é chamado de security association (SA - associação de segurança).

Descrição

Os campos sob o título “phase 1 proposal (authentication)” são os seguintes: Método de autenticação: mutual RSA. Meu identificador: meu endereço IP. Identificador do peer: endereço IP do peer. Meu certificado: structureality site-to-site VPN. Autoridade certificadora remota: structureality enterprise root. Os campos sob o título “phase 1 proposals (algorithms)” são os seguintes: Algoritmo de criptografia: AES-GCM de 256 bits com ICV de 128 bits. Algoritmo de hash: SHA 256. Grupo de chaves DH: 14 (2048 bits).

As negociações do IKE ocorrem em duas fases:

1. A fase I estabelece a identidade dos dois pares e realiza o acordo de chaves usando o algoritmo Diffie-Hellman para criar um canal seguro.
    
    Dois métodos de autenticação de pares costumam ser usados:
    
    - **Certificados digitais** — são emitidos a cada par por uma autoridade certificadora mutuamente confiável para que possam identificar um ao outro.
        
    - **Chave pré-compartilhada (pre-shared key)/autenticação de grupo)** : é quando a mesma senha é configurada para ambos os pares.
        
    
2. A fase II usa o canal seguro criado na fase I para estabelecer as cifras e os tamanhos de chaves que serão usados com o AH e/ou ESP na sessão de IPsec.

Há duas versões do IKE. A versão 1 foi desenvolvida para topologias site a site e host a host e requer um protocolo secundário para implementar VPNs de acesso remoto. O IKEv2 possui alguns recursos adicionais que o tornaram um protocolo popular a ser usado como uma solução independente de VPN de acesso remoto cliente a site. As principais alterações foram:

- Suporta métodos de autenticação EAP, permitindo, por exemplo, a autenticação do usuário em um servidor RADIUS.
- Oferece um modo de configuração simples que reduz a largura de banda sem comprometer a segurança.
- Permite a travessia de conversão de endereços de rede (NAT) e o multihoming MOBIKE. A travessia NAT facilita a configuração de um túnel permitido por um firewall/roteador doméstico. Com o multihoming, clientes que usam smartphone com interfaces Wi-Fi e celular podem manter a conexão IPsec ativa ao alternar entre elas.
