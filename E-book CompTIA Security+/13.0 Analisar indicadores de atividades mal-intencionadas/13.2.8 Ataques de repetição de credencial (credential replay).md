Um invasor pode estabelecer uma posição na rede comprometendo uma única estação de trabalho por meio de malware ou um ataque de senha. Uma vez que uma posição tenha sido conquistada, é provável que o objetivo do autor da ameaça seja identificar ativos de dados. Para isso, ele precisa encontrar formas de realizar movimentos laterais para comprometer outros hosts na rede e escalação de privilégios para obter mais permissões sobre os ativos da rede. Para atingir esses objetivos, além de quebrar mais senhas ou encontrar mais vulnerabilidades, ele pode usar [[credential replay attacks]] (ataques de repetição de credenciais).

Nota:

Em termos de ataques de rede, os credential replay attacks visam principalmente as redes do Windows Active Directory. Também existem credential replay attacks que visam aplicativos web. Discutiremos isso no próximo tópico.

Se uma conta de usuário em um host do Windows tiver sido autenticada em uma rede de domínio do Active Directory, o Serviço de Subsistema de Autoridade de Segurança Local (LSASS) armazenará vários segredos em cache na memória e no banco de dados de registro do Gerenciador de Contas de Segurança (SAM) para facilitar a autenticação única. Estes segredos incluem:

- Tíquete de concessão de tíquetes Kerberos (TGT) e chave de sessão. Isso permite que o host solicite tíquetes de serviço para acessar aplicativos.
- Tíquetes de serviço para aplicativos em que o usuário iniciou uma sessão.
- Hash NT de contas de usuário e serviço locais e no domínio que no momento estão conectadas de forma interativa ou remotamente na rede. As primeiras redes empresariais do Windows usavam a autenticação por desafio e resposta do NT LAN Manager (NTLM). Embora o protocolo NTLM esteja obsoleto para a maioria das finalidades, o hash NT ainda é usado como formato de armazenamento de credenciais. O hash NT é usado quando a autenticação NTLM herdada ainda é permitida e pode atuar na assinatura de solicitações e respostas Kerberos.

Crítico para a segurança da rede, se diferentes usuários estiverem conectados no mesmo host, as credenciais de todas essas contas podem ser armazenadas em cache pelo LSASS. Se algumas dessas contas forem de usuários mais privilegiados, como administradores de domínios, um autor de ameaças poderá usar as credenciais para ampliar os privilégios.

Nota:

O LSASS limpa os hashes da memória minutos após o usuário encerrar a sessão. O banco de dados SAM armazena em cache as credenciais de contas locais e da Microsoft, mas não as credenciais de domínio. Algumas edições do Windows implementam um recurso de virtualização chamado Credential Guard para proteger esses segredos de processos mal-intencionados, mesmo que tenham permissões do SYSTEM.

Os credential replay attacks usam vários mecanismos para obter e explorar esses segredos armazenados localmente para iniciar sessões autenticadas em outros hosts e aplicativos na rede. Por exemplo, se um autor de ameaças conseguir obter um hash NT, ele poderá usar um ataque pass the hash (PtH) para iniciar uma sessão em outro host se este estiver executando um serviço como compartilhamento de arquivos ou área de trabalho remota que ainda permita a autenticação NTLM.

Descrição

The process involves four steps as follows:

1. Victim logs on. O DC (Controlador de Domínio) verifica o usuário por meio do Kerberos.

2. A vítima faz login novamente. As credenciais Kerberos são armazenadas em cache na memória do processo LSASS.

3. O invasor extrai a memória do LSASS no computador da vítima. As credenciais em cache são reveladas.

4. O invasor utiliza o hash em outro computador. As credenciais em hash são reconhecidas pelo Kerberos.

A autenticação NTLM herdada fica em geral desativada, pois é um risco de segurança muito alto. Outros tipos de credential replay são direcionados contra a autenticação e autorização do Kerberos. Por exemplo, um ataque “golden ticket” tenta falsificar um tíquete de concessão de tíquete. Se for bem-sucedido, concede ao autor da ameaça acesso praticamente ilimitado a todos os recursos do domínio. Um ataque “silver ticket” tenta falsificar tíquetes de serviço. Ambos podem ser descritos como ataques pass the ticket (PtT).

A Microsoft lançou uma série de mitigações contra esses credential replay attacks específicos. Garantir que os hosts estejam totalmente atualizados e sigam bases de configuração seguras reduz muito sua eficácia. Quando ainda representam um risco, é possível configurar um sistema de detecção para correlacionar uma sequência de eventos de log de segurança, mas este método pode ser propenso a falsos-positivos. Antivírus e detecção de intrusão baseada em host geralmente conseguem detectar o código malicioso usado para extrair credenciais ou realizar ataques de falsificação de tíquetes.
