Ataques como repetição de sessão, CSRF e a maioria dos tipos de XSS são ataques do lado do cliente. Isso significa que eles executam um código arbitrário no navegador. Um ataque do [[lado do servidor]] faz com que o servidor faça algum processamento ou execute um script ou consulta de uma forma não autorizada pelo design do aplicativo. A maioria dos ataques do lado do servidor depende de algum tipo de ataque de injeção.

Um [[ataque de injeção]] explora alguma maneira insegura pela qual o aplicativo processa solicitações e consultas. Por exemplo, um aplicativo pode permitir que um usuário visualize seu perfil com uma consulta ao banco de dados que deveria retornar apenas o registro único correspondente ao perfil desse usuário. Um aplicativo vulnerável a um ataque de injeção pode permitir que um invasor obtenha os registros de todos os usuários ou altere campos no registro quando deveria apenas poder lê-los.

O XSS persistente e o abuso de consultas SQL e parâmetros discutidos anteriormente no curso são ambos tipos de ataque de injeção. Existem vários outros tipos de ataques de injeção que representam sérias ameaças às aplicações da Web e à infraestrutura.

## Injeção de linguagem de marcação extensiva

A [[Extensible Markup Language (Linguagem de marcação extensiva - XML)]] é usada por aplicativos para autenticação e autorização e para outros tipos de troca e upload de dados. Os dados enviados via XML sem criptografia ou validação de entrada são vulneráveis a spoofing (falsificação), falsificação de solicitação e injeção de dados ou códigos arbitrários. Por exemplo, um ataque de Entidade Externa XML (XXE) incorpora uma solicitação de um recurso local:

`<?xml version="1.0" encoding="UTF-8"?>`

`<!DOCTYPE foo [<!ELEMENT foo ANY ><!ENTITY bar SYSTEM "file:///etc/config"> ]>`

`<bar>&bar;</bar>`

Isso define uma entidade chamada bar que encaminha a um caminho de arquivo local. Um ataque bem-sucedido retornará o conteúdo de `/etc/config` como parte da resposta.

## Injeção de LDAP (Lightweight Directory Access Protocol)

O Lightweight Directory Access Protocol (LDAP) é outro exemplo de linguagem de consulta. O LDAP é usado especificamente para ler e gravar bancos de dados de diretório de rede. Um invasor pode explorar o acesso não autenticado ou uma vulnerabilidade em um aplicativo cliente para enviar consultas LDAP arbitrárias. Isso pode permitir que as contas sejam criadas ou excluídas, ou que o invasor altere autorizações e privilégios.

Os filtros LDAP são construídos a partir de pares de atributos (nome=valor) delimitados por parênteses e os operadores lógicos E (&) e OU (|). A adição de parâmetros de filtro como entrada não sanitizada pode contornar os controles de acesso. Por exemplo, se um formulário da Web for autenticado em um diretório LDAP com as credenciais válidas `Bob` e `Pa$$w0rd`, ele poderá construir uma consulta como esta a partir da entrada do usuário:

`(&(username=Bob)(password=Pa$$w0rd))`

Os dois parâmetros devem ser verdadeiros para que o login seja aceito. Se a entrada do formulário não for sanitizada, um invasor pode contornar a verificação de senha inserindo um nome de usuário válido mais uma string de filtro LDAP, como **bob)(&))**. Isso faz com que o filtro de senha seja descartado para uma condição que é sempre verdadeira:

`(&(username=Bob)(&))`
