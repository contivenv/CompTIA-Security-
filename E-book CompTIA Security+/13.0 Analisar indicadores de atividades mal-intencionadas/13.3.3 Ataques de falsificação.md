Ao contrário dos replay attacks, um [[ataque de falsificação]] sequestra uma sessão autenticada para executar alguma ação sem o consentimento do usuário.

## Falsificação de solicitação entre sites

Uma [[cross-site request forgery (falsificação de solicitação entre sites - XSRF - CSRF)]] pode explorar aplicativos que usam cookies para autenticar usuários e rastrear sessões. Para funcionar, o invasor deve convencer a vítima a iniciar uma sessão no site-alvo. Em seguida, ele passa uma solicitação HTTP ao navegador da vítima que falsifica uma ação no site de destino, como alterar uma senha ou um endereço de email. Essa solicitação pode ser disfarçada de maneiras que realizam o ataque sem que a vítima precise clicar em um link. Se o site de destino presumir que o navegador está autenticado porque há um cookie de sessão válido e não aplicar nenhum outro processo de autorização à entrada do invasor, ele aceitará a entrada como genuína. Isso também é chamado de ataque de confused deputy (representante confundido)

## Falsificação de solicitação do lado do servidor

Uma [[server-side request forgery (falsificação de solicitação do lado do servidor - SSRF)]] faz com que um aplicativo de servidor processe uma solicitação arbitrária que tem como alvo outro serviço. O serviço de destino pode ser outro aplicativo em execução no mesmo host ou um serviço em execução em um host remoto. A SSRF explora tanto a falta de autenticação entre os servidores internos quanto os serviços. Ela também depende de uma fraca validação de entrada, o que permite ao invasor enviar solicitações arbitrárias.

Os ataques de SSRF geralmente são direcionados à infraestrutura em nuvem, em que o servidor Web é apenas o componente voltado para o público de uma cadeia de processamento mais profunda. Uma aplicação Web típica é composta por várias camadas de servidores, com uma interface de cliente, camadas lógicas de intermediárias (middleware) e uma camada de banco de dados. As solicitações iniciadas a partir da interface do cliente (um formulário da Web) provavelmente exigirão várias solicitações e respostas entre o middleware e os servidores de back-end. Elas serão implementadas como solicitações de cabeçalho HTTP e respostas entre cada servidor. A SSRF é um meio de acessar esses servidores internos, fazendo com que o servidor público execute solicitações neles. Enquanto na CSRF uma exploração tem apenas os privilégios do cliente, com SSRF a solicitação manipulada é feita com o nível de privilégio do servidor.
