Os ataques de sequestro/reprodução de sessão, falsificação e injeção são difíceis de identificar, mas os pontos de partida para a detecção provavelmente serão a análise do URL e o log de acesso do servidor Web.

## Análise do localizador padrão de recursos

Além de apontar para o host ou local de serviço na Internet (por nome de domínio ou endereço IP), um [[Uniform Resource Locator (Localizador padrão de recursos - URL)]] pode codificar alguma ação ou dados a serem enviados ao host do servidor. Este é um vetor comum para atividades maliciosas.

Como parte da análise do URL, é importante entender como o HTTP funciona. Uma sessão HTTP começa com um cliente (um agente de usuário, como um navegador da Web) fazendo uma solicitação a um servidor HTTP. A conexão estabelece uma conexão TCP. Essa conexão TCP pode ser usada para várias solicitações ou um cliente pode iniciar novas conexões TCP para diferentes solicitações. Uma solicitação normalmente compreende um método, um recurso (como um caminho de URL), número da versão, cabeçalhos e corpo. Os principais métodos são os seguintes:

- **GET**: recuperar um recurso.
- **POST** (enviar): enviar os dados para o servidor para processamento pelo recurso solicitado.
- **PUT** (colocar): riar ou substituir o recurso.

Os dados podem ser enviados a um servidor usando um método POST ou PUT e os cabeçalhos e corpo HTTP, ou codificando os dados dentro do URL usado para acessar o recurso. Os dados enviados por meio de uma URL são delimitados pelo caractere ?, que aparece após o caminho do recurso. Em geral, os parâmetros de consulta são formatados como um ou mais pares nome=valor, com e comercial (&) delimitando cada par.

A resposta do servidor compreende o número da versão e um código de status e mensagem, além de cabeçalhos opcionais e corpo da mensagem. Um código de resposta HTTP é o valor do cabeçalho retornado por um servidor quando um cliente solicita um URL, como 200 para “OK” ou 404 para “Não encontrado”.

## Codificação por cento

Um URL pode conter apenas caracteres não reservados e reservados do conjunto padrão. Os caracteres reservados são usados como delimitadores dentro da sintaxe do URL e só devem ser usados de forma não codificada para esses fins. Os caracteres reservados são os seguintes:

`: / ? # [ ] @ ! $ & ' ( ) * + , ; =`

Também existem caracteres não seguros, que não podem ser usados em um URL. Caracteres de controle, como terminação de string nula, mudança de linha, avanço de linha, fim de arquivo e tabulação, não são seguros. A codificação percentual permite que um user-agent (agente do usuário) envie qualquer caractere seguro ou inseguro (ou dados binários) ao servidor dentro do URL. Seus usos legítimos são codificar caracteres reservados dentro do URL quando eles não fazem parte da sintaxe do URL e enviar caracteres Unicode. A [[codificação percentual]] pode ser usada indevidamente para ofuscar a natureza de um URL (codificação de caracteres não reservados) e enviar entradas maliciosas.
