Um [[ataque de aplicativo]] visa uma vulnerabilidade no sistema operacional ou no software de aplicativo. Uma vulnerabilidade no aplicativo é uma falha de design que pode abrir lacunas no sistema de segurança do aplicativo ou que leve o aplicativo falhar. Existem, em geral, dois cenários principais para ataques a aplicativos:

- Comprometer o sistema operacional ou aplicativos de externos em um host de rede explorando cavalos de Tróia, anexos maliciosos ou vulnerabilidades do navegador. Isso permite que o autor da ameaça obtenha uma base de apoio em uma rede local.
- Comprometer a segurança de um site ou aplicativo web. Isso permite que um autor de ameaças obtenha o controle de um host web e roube dados dele ou use-o para tentar penetrar ainda mais na rede.

O aumento do número de falhas e erros de aplicativos pode fornecer um indício geral de que um autor de ameaças está tentando explorar uma vulnerabilidade em um serviço de rede, sistema operacional ou aplicativo de desktop ou aplicativo web. Os erros podem ser registrados em um log do sistema ou em um log específico do aplicativo, dependendo da natureza do software e do tipo de evento de falha. A utilização anômala da CPU, memória, armazenamento ou rede também pode ser um indício de ataque ao aplicativo. No entanto, esses indícios também podem ter várias causas não mal-intencionadas, por isso é importante correlacioná-los a fatores que identificam tipos específicos de ataques a aplicativos.

## Escalação de privilégio

O objetivo da maioria dos ataques a aplicativos é permitir que o autor da ameaça execute seu próprio código no sistema. Isso é chamado de [[execução de código arbitrário]]. Quando o código é transmitido de uma máquina para outra, denomina-se execução [[remota de código]]. O código normalmente seria desenvolvido para instalar algum tipo de backdoor ou para desativar o sistema de alguma forma.

Um aplicativo ou processo deve ter privilégios para ler e gravar dados e executar funções. Dependendo de como o software registra gravações, um processo pode ser executado usando uma conta do sistema, a conta do usuário conectado ou uma conta designada. Se uma exploração de software funcionar, o invasor poderá executar um código arbitrário com o mesmo nível de privilégio do processo explorado. Existem dois tipos principais de [[escalação de privilégios]]:

- A [[escalação vertical de privilégios]] é quando um usuário ou aplicativo conseguem acessar funcionalidades ou dados que não deveriam estar disponíveis para eles. Por exemplo, um processo pode ser executado com privilégios de administrador local, mas uma vulnerabilidade permite que o código arbitrário seja executado com privilégios de SYSTEM mais elevados.
- A [[escalação horizontal de privilégios]] é quando um usuário acessa funcionalidades ou dados destinados a outro usuário. Por exemplo, por meio de um processo em execução com privilégios de administrador local em uma estação de trabalho do cliente, o código arbitrário pode ser executado como uma conta de domínio em um servidor de aplicativos.

Sem realizar uma análise detalhada do código ou da execução do processo em tempo real, é a escalação de privilégios que dá o indicador mais simples de um ataque a aplicativos. Se o registro em log dos processos tiver sido configurado, o log de auditoria poderá fornecer evidências de tentativas de escalação de privilégios. Estas tentativas também podem ser detectadas por agentes de resposta a incidentes e de proteção de endpoint, que exibirão um alerta.

## Buffer overflow

Buffer é uma área da memória que um aplicativo reserva para armazenar algum valor. O aplicativo espera que os dados estejam em conformidade com algum tamanho ou formato de valor esperado. Para explorar uma vulnerabilidade de buffer overflow, o invasor passa dados que preenchem deliberadamente o buffer até o fim e, em seguida, sobrescreve os dados no início. Uma das vulnerabilidades mais comuns é um stack overflow. Stack é uma área da memória usada por uma sub-rotina de programa. Inclui um endereço de retorno, que é o local do programa que chamou a sub-rotina. Um invasor pode usar um buffer overflow para alterar o endereço de retorno, o que lhe permitirá execute um código arbitrário no sistema.

Os sistemas operacionais usam mecanismos como randomização de layout do espaço de endereçamento (ASLR) e prevenção de execução de dados (DEP) para mitigar os riscos decorrentes de buffer overflow. As tentativas fracassadas de buffer overflow podem ser identificadas por meio de falhas frequentes do processo e outras anomalias.
