O sistema de nomes de domínio (DNS) resolve solicitações de hosts e serviços nomeados para endereços IP. A resolução de nomes é um método de endereçamento crítico na internet e em redes privadas. Há muitos ataques potenciais contra o DNS. Na internet pública, os ataques podem usar técnicas de typosquatting para fazer com que as vítimas confundam sites maliciosos com sites legítimos. O DNS pode ser explorado em um ataque DRDoS. Os autores de ameaças também podem alvejar diretamente os serviços de DNS público como um meio de realizar DoS contra um site ou recurso de nuvem. Por fim, um autor de ameaças pode conseguir desviar um servidor DNS público e inserir registros falsificados, direcionando as vítimas para sites mal-intencionados.

Em uma rede privada, é provável que um ataque DNS envolva algum tipo de [[envenenamento de DNS]]. O envenenamento de DNS compromete o processo pelo qual os clientes consultam os servidores de nomes para localizar o endereço IP de um nome de domínio. Existem várias maneiras de realizar um ataque de envenenamento de DNS.

## Ataques on-path baseados em DNS

Se o autor da ameaça tiver acesso à mesma rede local da vítima, ele pode usar o envenenamento de ARP para responder a consultas DNS da vítima com respostas falsificadas. Isso pode ser combinado com um ataque de negação de serviço no servidor DNS legítimo da vítima. Um DHCP falso pode ser usado para configurar clientes com o endereço de um resolvedor de DNS controlado pelo autor da ameaça.

## Envenenamento de cache do cliente DNS

Antes do desenvolvimento do DNS na década de 1980, a resolução de nomes ocorria usando um arquivo de texto chamado HOSTS. Cada mapeamento de nome:endereço IP era registrado nesse arquivo, e os administradores do sistema tinham que baixar a cópia mais recente e instalá-la manualmente em cada cliente ou servidor da internet. Embora a maioria da resolução de nomes agora funcione por meio do DNS, o arquivo HOSTS ainda está presente e a maioria dos sistemas operacionais o verifica antes de usar o DNS. Seu conteúdo é carregado em um cache de mapeamentos conhecidos de nome:endereço IP, e o cliente só entra em contato com um servidor DNS se o nome não estiver em cache. Portanto, se um invasor conseguir colocar um mapeamento falso de nome:endereço IP no arquivo HOSTS e efetivamente envenenar o cache DNS, ele poderá redirecionar o tráfego. A modificação do arquivo HOSTS requer acesso de administrador. Nos sistemas UNIX e Linux, ele é armazenado como `/etc/hosts`, e no Windows é colocado em `%SystemRoot%\System32\Drivers\etc\hosts`. A presença de entradas suspeitas no arquivo HOSTS é um indício de que a máquina foi comprometida.

## Envenenamento de cache do servidor DNS

O envenenamento de cache do servidor DNS visa corromper os registros mantidos pelo próprio servidor DNS. É possível fazer isso executando o DoS contra o servidor que contém os registros autorizados para o domínio e, em seguida, falsificando respostas a solicitações de outros servidores de nomes. Outro ataque do tipo leva o servidor de nome da vítima a responder a uma consulta recursiva do host invasor. Uma consulta recursiva obriga o servidor DNS a consultar o servidor autoritativo para obter a resposta em nome do cliente. O invasor disfarça seu DNS como um servidor de nomes autoritativo e, ao responder à consulta, replica a resposta original, mas também inclui diversos mapeamentos falsos de domínio:IP para outros domínios, que o DNS da vítima acaba aceitando como legítimos. A ferramenta `nslookup` ou `dig` pode ser usada para consultar os registros de nome e os registros em cache mantidos por um servidor, a fim de descobrir se foram inseridos registros falsos.

## Indicadores de ataque DNS

Um servidor DNS pode registrar um evento cada vez que processa uma solicitação para fazer a conversão entre um nome de domínio e um endereço IP. Os logs de eventos DNS podem conter uma variedade de informações que podem fornecer informações de segurança úteis e indícios de ataque, como os seguintes:

- Os tipos de consultas que um host fez ao DNS.
- Os hosts que estão em comunicação com endereços IP ou domínios suspeitos.
- Anomalias estatísticas, como picos ou um número consistentemente grande de falhas de pesquisa de DNS, que podem apontar para computadores infectados com malware, configurados incorretamente ou executando aplicativos obsoletos ou defeituosos.

Nota:

O DNS também é uma escolha popular para implementar o comando e controle (C&C) de cavalos de Troia de acesso remoto. Ele pode ser usado como um meio de exfiltrar secretamente dados de uma rede privada.
